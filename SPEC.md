# sekai_reader — 프로젝트 스펙

> 이 문서는 세션 간 인계를 위한 단일 참조 문서다. 새 세션은 작업 시작 전 이 문서를 먼저 읽는다.
> 최초 작성: 2026-06-22

## 1. 목표

프로젝트 세카이(프세카)를 BlueStacks(에뮬레이터)로 Mac에서 구동하면서,
**AI/알고리즘이 화면을 스스로 인식 → 미열람(skip 표시) 스토리를 찾아냄 → 해당 스토리에 진입 →
(skip 버튼이 아니라) 직접 빠르게 연타로 진행 → 스토리의 "미열람(skip) 표시"를 제거**하는
자동화 도구를 만든다.

- 최종 결과물 형태: **데스크탑 앱** (시작/정지, 진행상황 표시 등)
- 사용자 의향: **"결과만 나오면 됨"** (자체 AI 모델 학습은 필수 아님). 단, **토큰 비용 최소화**가 중요.

## 2. 환경 (실측 확인됨, 2026-06-22)

| 항목 | 상태 |
|---|---|
| OS | macOS (Apple Silicon, Darwin 25.5.0) |
| 에뮬레이터 | **BlueStacks Air** (Apple Silicon판). `BlueStacks.app`, `BlueStacks Air multi-instance manager.app` 설치됨 |
| 번들 ADB | `/Applications/BlueStacks.app/Contents/MacOS/hd-adb` (별도 adb 미설치) |
| 디바이스 | `emulator-5554` (model SM-G998B, 화면 폭 1080px 추정) |
| Python | 3.14.2 (`/opt/homebrew/bin/python3`) |
| Node | v22.22.1 |
| Tesseract(OCR) | 설치됨 (`/opt/homebrew/bin/tesseract`) |
| 프세카 설치 | **아직 안 됨** (QooApp/Softonic에서 받는 중이었음). 패키지 목록에 세카이 없음 |

### ⚠️ 가장 중요한 환경 제약 — 실측 결과

이 BlueStacks Air 환경에서 ADB 동작이 비대칭이다:

| ADB 동작 | 결과 |
|---|---|
| 화면 캡처 `exec-out screencap -p` | ✅ **5/5 성공 (완벽, 안정적)** |
| 탭 입력 `input tap` (`shell` / `exec-out` 둘 다) | ❌ **0/8 실패** (`error: closed`) |
| `wm size`, `pm list packages` | ❌ 끊김 (`error: closed`) |
| `getprop` | ⚠️ 간헐적으로만 됨 |

**원인 추정**: BlueStacks Air는 네이티브 바이너리(screencap)는 ADB로 허용하지만,
안드로이드 런타임/binder를 거치는 명령(input/pm/wm/am 등)은 ADB 경로에서 막혀 있다.

**결론 (설계에 직결)**:
- **눈(화면 읽기)** → ADB `screencap`으로 해결 ✅
- **손(탭/연타)** → ADB 불가 → **macOS 차원에서 BlueStacks 창을 직접 마우스 클릭**해야 함
  (cliclick 또는 Python Quartz/pyautogui + "안드로이드 픽셀 좌표 → macOS 화면 좌표" 변환 필요)

### ADB 사용 팁 (실측에서 얻은 것)
- 같은 에뮬레이터가 `emulator-5554`와 `127.0.0.1:5555`로 **중복 등록**되면 충돌하므로 하나로 고정.
- `exec-out`은 셸을 안 거치고 바이너리를 직접 실행하므로, `exec-out "wm size; echo x"`처럼
  셸 문법(`;`, 파이프)을 넘기면 실패한다. 셸 기능이 필요하면 `shell`을 써야 하지만 이 환경선 `shell`도 불안정.
- 캡처 명령(확정 동작): `/Applications/BlueStacks.app/Contents/MacOS/hd-adb -s emulator-5554 exec-out screencap -p > out.png`

## 3. 아키텍처

```
[ADB screencap]  → 화면 PNG 캡처 (안정적, 토큰 0)
       ↓
[인식 레이어]    → "미열람 스토리 어디?" / "지금 무슨 화면?" / "스토리 끝났나?"
       ↓
[좌표 변환]      → 안드로이드 픽셀 좌표 → macOS 화면(BlueStacks 창) 좌표
       ↓
[macOS 마우스]   → 클릭/연타 (cliclick 또는 Python Quartz/pyautogui)
```

- 언어: **Python** (opencv-python, pillow, pytesseract, anthropic SDK 등)
- 데스크탑 앱 셸: 추후 결정 (PySide/Tkinter 등 Python GUI, 또는 Electron/Tauri)

## 3.6 실측: 사이드 스토리 화면 흐름 (2026-06-22, JP판 랭크353 계정)

게임 설치/실행 확인 완료. **서버 = JP 확정.** ADB 연결: `hd-adb connect 127.0.0.1:5555` (5554 말고 5555 포트가 LISTEN). 5555에서도 `input tap`/`wm size`는 여전히 `error: closed` → **입력은 ADB 불가 재확인.** 캡처는 1920x1080 PNG로 안정적.

자동화 대상 = **사이드 스토리**. 네비게이션 단계:

1. **홈** 하단 `ストーリー` → 2. **스토리 선택**(메인/이벤트 vs **サイドストーリー**) → 3. **그룹 선택**(왼쪽 세로 탭 6개) → 4. **캐릭터 선택**(가운데 카드) → 5. **에피소드 목록**.

### 그룹·캐릭터 (총 26명)
| 그룹 | 캐릭터 수 |
|---|---|
| VIRTUAL SINGER | 6 (ミク/リン/レン/ルカ/MEIKO/KAITO) |
| Leo/need | 4 |
| MORE MORE JUMP! | 4 |
| Vivid BAD SQUAD | 4 |
| ワンダーランズ×ショウタイム | 4 |

- 그룹 탭·캐릭터 카드에는 **미열람 마커가 없음** → 어느 캐릭터에 안 본 게 있는지 미리 알 수 없음 → **전 캐릭터 순회**가 기본 전략.
- 캐릭터 카드 위치는 인원수에 따라 재배치됨 → 좌표 박지 말고 동적으로(템플릿/개수기반) 잡아야 함.

### 에피소드 목록 화면 (핵심)
- 한 에피소드 = **`前編`(전편) / `後編`(후편)** 버튼 2개 (각각 별개로 읽음).
- 목록은 **길어서 스크롤 필요**. 좌/우 2열 배치.
- 버튼 3가지 상태:
  - 🟢 **초록 `SKIP` 말풍선 뱃지**(버튼 우상단) = **미열람**. ← 우리가 찾아 없앨 대상.
  - 🔒 **남색 자물쇠** = 캐릭터 랭크 부족으로 **잠김**(Lv.5/30/60 등). **읽기 불가 → 스킵.**
  - 뱃지·자물쇠 둘 다 없음 = **이미 읽음.**
- 원본/템플릿: `assets/captures/sidestory_list_miku.png`, `assets/templates/_raw_skip_and_lock.png`

### 1편 진행 흐름 (실측, 선택지·중간팝업 없음)
1. SKIP 뱃지 붙은 `前編` 클릭 → **보이스 다운로드 다이얼로그** 뜸:
   - "ボイスデータをダウンロードして、ストーリーを読みますか?" / 버튼 `キャンセル | ボイスなし | ボイスあり`
   - **자동화는 `ボイスなし`(약4.7MB) 선택**(빠르고 가벼움). 템플릿: `assets/captures/voice_download_dialog.png`
2. 진입 → **스토리 재생 화면**: 하단 대사창, 우상단 `・・・`, 우하단 `♪`.
   - **화면 중앙 아무 데나 탭하면 다음 대사로 진행** → **중앙 안전지대 연타**. (우상단·우하단 피함)
   - 인게임 SKIP 버튼은 안 씀(SPEC 목표대로 연타로 통과). 템플릿: `assets/captures/story_playback.png`
3. 끝까지 연타 → **선택지/팝업 없이** 자동으로 **에피소드 목록으로 복귀**. 해당 버튼의 SKIP 뱃지 사라짐(=미열람 제거 확인). 목록은 다른 에피소드의 SKIP 뱃지를 계속 보여줌.

### ✅ "손"(입력) 해결 — macOS 마우스 클릭 경로 확립 (2026-06-22)
ADB input 불가 → **cliclick + 좌표 변환**으로 해결. 실제 클릭으로 게임 반응까지 검증 완료
(後編 SKIP 버튼 클릭 → 보이스 다이얼로그 등장 → キャンセル 클릭 → 목록 복귀, 왕복 성공).

- **클릭 수단 = cliclick** (`brew install cliclick`). 손쉬운 사용(Accessibility) 권한 필요(부여됨).
- **화면 기록 권한**: macOS 스샷(보정용)에 필요. 게임 읽기는 ADB라 평소엔 불필요. `/remote-control`로 부여함.
- **창**: "BlueStacks Air", 실측 (15,87) 1632x932 pt. Retina ×2 (스샷은 3264x1864 px).
- **좌표 변환 (보정값, SPEC핵심)**: 창 안 게임 렌더 영역 = 상단 타이틀바 ~32pt, 우측 툴바 ~31pt 제외, 좌/하단 flush. 멀티스케일 템플릿매칭 보정 점수 0.995.
  ```
  click_x_pt = win_x + (ax/1920)*(win_w - 31)
  click_y_pt = win_y + 32 + (ay/1080)*(win_h - 32)
  ```
  창 위치/크기는 런타임에 osascript로 읽음 → 창 이동해도 동작. ⚠️ 단, 창을 **리사이즈하면 크롬 비율 가정이 깨질 수 있음**(타이틀바/툴바가 고정 pt라는 전제) → 재보정 필요.

### 구현된 코드 (2026-06-22)
- `sekai/capture.py` — ADB screencap → cv2 BGR ndarray (`grab()`). 디바이스 `127.0.0.1:5555`.
- `sekai/input.py` — `window_geometry()`, `android_to_screen()`, `tap(ax,ay)`, `swipe(...)` (cliclick).
  - **스크롤도 macOS 드래그로 해결**(ADB swipe도 막힘). 실측: (960,820)→(960,300) 드래그 1회 ≈ 콘텐츠 ~400px(≈2행) 스크롤, 관성 있음 → 정확 px 의존 말고 '스크롤→캡처→안 변하면 바닥' 루프 사용.
- `assets/templates/skip_badge.png` — 초록 SKIP 뱃지 템플릿(89x42). adb 캡처에서 0.96으로 검출 확인.
- `assets/captures/` — 참조 스샷(목록/다이얼로그/재생).
- `.venv` (python3.14) — opencv-python-headless, numpy, pillow 설치됨.

### 인식·루프 (구현+검증 완료, 2026-06-22)
- `sekai/recognize.py`:
  - `count_buttons()`/`is_list_screen()` — 흰 알약버튼 개수(목록 8~13, 재생/다이얼로그 0~1, 임계 ≥3). 종료감지용.
  - `find_skip()` — 초록 SKIP 뱃지 전수 검출. `skip_targets()` — 뱃지→가장 가까운 흰버튼 중심(=클릭좌표). 前編/後編 자동 구분.
  - `find_voice_none()` — 보이스 다이얼로그 'ボイスなし' 버튼 (≈(940,965)).
- `sekai/reader.py` `read_one(target)`: 진입→ボイスなし→중앙(960,540) 연타→`is_list_screen` 2회연속이면 종료. 연타지점은 목록의 칼럼 사이 빈공간이라 오버탭해도 무해.
- 템플릿: `assets/templates/skip_badge.png`(89x42), `voice_none.png`(250x74).

### 남은 작업 = 순회 오케스트레이션 (로드맵 5)
- 목록 **전체 스캔**: 맨위로 → (캡처→skip_targets 클릭→read_one→복귀) → swipe로 스크롤 → 화면 안 변하면 바닥. 스크롤 후 좌표 재계산 필수(스크롤로 y 바뀜).
- **캐릭터/그룹 네비게이션**: 캐릭터 선택(개수 가변 6 or 4) → 사이드스토리 → 처리 → 뒤로(←) → 다음 캐릭터/그룹. 그룹탭 좌측 세로, 캐릭터 카드 가운데.
- 데스크탑 앱 포장(시작/정지/진행률) = 로드맵 6.

## 4. 인식 전략 — "로컬 우선, AI는 보험" (토큰 최소화 핵심)

대부분을 **로컬 알고리즘(공짜)**으로 처리하고, Claude는 막힐 때만 호출한다.

| 작업 | 수단 | AI? | 토큰 |
|---|---|---|---|
| 스토리 안에서 빠르게 연타 | 안전 좌표 반복 클릭 | ❌ | 0 |
| "스토리 끝났나?" 판단 | OpenCV 템플릿 매칭(특정 버튼 등장 감지) | ❌ | 0 |
| 목록에서 미열람 마커 찾기 | OpenCV 템플릿 매칭 | ❌ | 0 |
| 메뉴 글자 읽기 | Tesseract OCR | ❌ | 0 |
| 화면이 예상과 다른 애매한 상황 | **Claude Vision 호출** | ⚠️ 가끔 | 극소량 |

- **OpenCV** = 학습하는 AI가 아니라 **고정된 이미지 비교 알고리즘**(템플릿 매칭 = 숨은그림찾기). 결정적·공짜.
- 프세카는 **UI가 고정**이라 로컬 템플릿 매칭이 잘 먹힌다 → 자체 모델 학습 불필요.
- 매 프레임을 LLM에 보내면 비싸지지만(초당 1장 1시간 ≈ 600만 토큰), 이 설계는 LLM 호출이 거의 0이다.

## 7. 용어 메모 (비개발자 참고용)

- **ADB**: 안드로이드를 컴퓨터에서 명령으로 조작하는 도구.
- **input tap x y**: ADB로 (x,y) 좌표를 손가락처럼 탭하는 명령. ← 이 환경선 막힘.
- **screencap**: ADB로 현재 화면을 이미지로 캡처. ← 이 환경선 잘 됨.
- **OpenCV**: 화면에서 특정 모양을 찾는 공짜 알고리즘 도구(학습 X, 결정적).
- **템플릿 매칭**: 미리 잘라둔 작은 그림 조각을 큰 화면에서 찾아 좌표 반환 (숨은그림찾기).
- **OCR / Tesseract**: 화면 속 글자를 텍스트로 읽어내는 도구.
- **Claude Vision**: 스샷을 보내 "무슨 화면이냐" 판단시키는 진짜 AI (비용 발생, 비상용).
